# SpecViewer — デザイン仕様書

設計書を閲覧するためのサイト「SpecViewer」のデザイン仕様・決定事項。
設計検討を通じて確定した内容を逐次反映する。

---

## 1. 概要

| 項目 | 内容 |
|---|---|
| サイト名 | SpecViewer |
| 目的 | 設計書（仕様書）を閲覧・検索・ナビゲートする |
| 技術スタック | VitePress 2.0 + Tailwind CSS v4 (+ Vue) |
| ソース | `docs/` を `srcDir` とする Markdown（git 管理＝版管理） |
| 想定利用者 | 社内メンバー |
| 想定規模 | 文書数百件程度 |

## 2. スコープ

**閲覧専用（Viewer）。** 読む・探す・ナビゲートするのみを担う。

- 編集・新規作成 → GitHub 等の外部で行う
- 承認・公開ワークフロー → 持たない
- 認証・権限制御 → 持たない（静的サイト）

### 2.1 除外したもの（決定済）

- 改訂履歴（revision history）— 2026-06-14 削除決定
- 新規作成・編集画面
- 承認・公開ワークフロー
- ロール・権限制御

## 3. 情報アーキテクチャ（IA）

**第1階層 = 3 セクション**（グローバルナビに直接対応）:

| セクション | 扱う内容 |
|---|---|
| **API** | API仕様書・エンドポイント定義・スキーマ |
| **画面** | 画面設計書・UI仕様・画面遷移 |
| **ガイド** | 執筆ガイドライン・テンプレート・規約 |

セクションとディレクトリ（URL）の対応（確定）:

| セクション | ディレクトリ | URL |
|---|---|---|
| API | `docs/api/` | `/api/` |
| 画面 | `docs/ui/` | `/ui/` |
| ガイド | `docs/guide/` | `/guide/` |

> 「画面」セクションの URL 表現は `ui`（画面設計 = UI）。日本語「画面」は URL に不適なため。
> 旧候補（アーキテクチャ / データ / セキュリティ / 運用）は、この3分類の**下位（サブカテゴリ or タグ）**に回す。

### 3.1 配下構造（決定: ② サブカテゴリツリー）

`セクション → サブカテゴリ（多階層）→ 文書` の **多階層ツリー**。
文書規模は数百件程度を想定するため、フラットではなくツリーを採用。
section は **ディレクトリ第1階層**（`docs/api/` 等）、サブカテゴリは **frontmatter の `subcategory`** で表現する（英 directory + 日本語表示のため）。
`subcategory` は **文字列1つ または 配列** で指定でき、配列にすると多階層ネストになる（例: `[認証, トークン]`）。文字列1つは `[認証]` と同義（後方互換）。

## 4. データモデル（frontmatter）

設計書は本文 Markdown と同ファイルの frontmatter にメタ情報を集約する。
**一意性はファイルパス（URL）で表す** — `id` の採番は行わない。

```yaml
---
title: 認証・認可アーキテクチャ
description: OAuth 2.1 / OIDC を用いた…   # 一覧・OGP で使用
subcategory: 認証           # サブカテゴリ。文字列1つ または 配列で多階層（例: [認証, トークン]）
status: 承認済み            # ドラフト | レビュー中 | 承認済み | 廃止
dependsOn:                 # 依存している文書（方向的・docs/ からの path 配列）
  - guide/architecture-overview
related:                   # 関連文書（参照用・docs/ からの path 配列）
  - api/auth/signup
---
```

> **注:** `id` / `author` / `tags` / `category` / `reviewers` / `created` は廃止 — 識別子はファイルパス、section はディレクトリから導出。`revision` / 改訂履歴も削除済み。
> `dependsOn` = この文書が依存する文書（方向的）。`related` = 関連文書（参照用）。ともに docs/ からの相対 **path**（.md 無し）で指定する（title の揺らぎでリンク切れしない・エディタのパス補完が効く）。
> section はディレクトリ第1階層から導出、`subcategory` は frontmatter で指定する（英 directory + 日本語表示のため）。文字列1つまたは配列で多階層ネストを表現できる。
> 更新日は frontmatter に書かず、**git の最終コミット日時**から自動取得する（`config.lastUpdated: true`）。

## 5. デザイン方向

**採用: 01 Reference**（クリーンなライト系ドキュメント）

- タイポグラフィ: Inter
- アクセント: indigo（`#4f46e5`）
- ニュートラル: slate
- 角丸: `rounded-xl`、余白多め、細いボーダー

> 初期検討では公文書風（RFC）・ターミナル風・ミニマル風も比較検討したが、可読性と汎用性から Reference を採用（比較案のモックアップは削除済み）。

### 5.1 ダークモード（確定）

VitePress の `appearance` + Tailwind v4 の `@custom-variant dark`（`.dark` クラス方式）で実装。ヘッダーのトグル（月↔太陽）で切替。OS 設定を既定とし、手動切替は localStorage に保存・フラッシュ防止 inline script 付き。

配色マッピング（ライト → ダーク）:

| 用途 | ライト | ダーク |
|---|---|---|
| ページ背景 | `bg-white` | `bg-slate-900` |
| ヘッダー背景 | `bg-white/85` | `bg-slate-900/85` |
| カード/hover 背景 | `bg-slate-50` / `bg-slate-100` | `bg-slate-800`（hover は `/50`） |
| 本文テキスト | `text-slate-700` | `text-slate-300` |
| 見出し | `text-slate-900` | `text-slate-100` |
| 補助 / 弱 | `text-slate-500` / `text-slate-400` | `text-slate-400` / `text-slate-500` |
| ボーダー / 区切り | `border-slate-200/100` / `divide-slate-100` | `border-slate-800` / `divide-slate-800` |
| brand テキスト | `text-brand-600` / `text-brand-700` | `text-brand-400` / `text-brand-300` |
| brand バッジ背景 | `bg-brand-50` | `bg-brand-500/10` |
| ステータス（emerald/amber/sky） | `bg-*-50 text-*-600/700` | `bg-*-500/10 text-*-400` |

アクセント（`brand-600` 背景・ステータスドット `bg-*-500`）はライト/ダーク共通で維持。

## 7. サイドバー仕様（確定）

**コンテキスト型セクションツリー（多階層）** — 現セクションの中身のみを表示。

| 項目 | 内容 |
|---|---|
| 種別 | 現セクション（API/画面/ガイド）の中身のみ。ヘッダーでセクション切替 |
| 構造 | セクション → サブカテゴリ（多階層・各層で折たたみ ▾）→ 文書 |
| 配置 | section ページ・doc ページのみ。**ホームには置かない**（フルワイド） |
| 役割 | 構造ナビ専用。**フィルタ・ソートは持たない**（→ コンテンツ上部ツールバー） |
| 現在地 | 現在文書をハイライト |
| モバイル | ドロワー（ハンバーガーで開閉） |
| **全体折りたたみ** | サイドバー上部 « で隠す ／ `Ctrl/Cmd+B` で切替。隠した際は左端に » 復元ボタン。状態は保持（localStorage） |

```
┌──────────────────────────┐
│ API                      │ ← 現セクション（sticky）
│──────────────────────────│
│ ▾ 認証                    │
│   スコープ設計              │ ← 直下文書
│   ▾ トークン               │ ← サブサブカテゴリ（多階層）
│     トークン仕様            │ ← 現在（ハイライト）
│ ▾ 注文                    │
│   注文 API 仕様             │
│   注文状態遷移              │
│ ▸ 決済                    │ ← 折たたみ
│──────────────────────────│
│ 画面 ↗   ガイド ↗         │ ← 他セクション（任意）
└──────────────────────────┘
```

**サブカテゴリの取得:** frontmatter の `subcategory`（文字列1つ / 配列）から `subcategoryTree()` で自動構築。配列要素の順序で多階層ツリーを生成する（再帰コンポーネント `SidebarTree.vue`）。

## 13. API設計フォーマット

「API」セクションの設計書（原則 1文書＝1エンドポイント）は [`docs/guide/api-format.md`](docs/guide/api-format.md) の標準フォーマットに従う。規範サンプル: [`docs/api/auth/signup.md`](docs/api/auth/signup.md)（`POST /signup`）。

- 構成: **API情報 / リクエスト / レスポンス / 副作用 / シナリオ(BDD/Gherkin)**
- **OpenAPI は使わず** Markdown のみ（人間可読・軽量）
- エラー形式: `{ code, message }`（コードは `UPPER_SNAKE_CASE`）
- **ページネーションは標準化せず案件毎**
- 副作用は主操作と成否を分離（最終的整合性）・idempotency key で冪等

## 14. 大カテゴリ（セクション）の追加手順

新しい大カテゴリ（例: 運用）を追加する手順。3箇所の追記で全画面に反映される。

1. **`docs-data.ts`** — `SectionKey` 型と `sections` 配列に追加:
   ```ts
   export type SectionKey = 'api' | 'ui' | 'guide' | 'ops'   // 型に追記

   // sections 配列にメタを追加
   {
     key: 'ops',
     label: '運用',
     description: '運用・監視・デプロイに関する設計書',
     shortDescription: '運用・監視・デプロイ',
     navHref: '/ops/',
     iconClass: 'bg-orange-50 text-orange-600 dark:bg-orange-500/10 dark:text-orange-400',
     badgeClass: 'bg-orange-50 text-orange-600 dark:bg-orange-500/10 dark:text-orange-400',
     badgeText: '運',
     accentText: 'text-orange-600 dark:text-orange-400',
   }
   ```
2. **`SectionIcon.vue`** — 新しい key のアイコン SVG を `v-if`/`v-else-if` チェインに1件追加
3. **`docs/{key}/index.md`** を作成:
   ```yaml
   ---
   layout: section
   section: ops
   title: 運用
   ---
   ```

これでホーム（セクションカード）・グローバルナビ・セクション一覧・サイドバー・検索・依存マップに**自動反映**される（`deriveSection` は `sections` から動的に判定するため追記不要）。文書は `docs/ops/.../*.md` を置くだけ。
